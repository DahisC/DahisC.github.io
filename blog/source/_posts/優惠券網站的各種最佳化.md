# 最佳化

## 減少用戶的請求優惠券資料的次數

> 網站上儲存的優惠券數量大約是 1000 張左右，但因為每張優惠券除了本身搭配的餐點之外，每個餐點也都會有其餘可選的選項，平均每張優惠券的 JSON 大小大約是 20KB

此處使用了 Pinia & Pinia Persisted State 插件將優惠券儲存在 Local Storage，搭配伺服器每天更新兩次資料庫並記錄更新時間，使用者每次打開網站會先比對本地與資料庫的更新時間，如果不一樣才重新從伺服器撈取 

## 單一頁面顯示太多優惠券導致渲染速度過慢

1. 在前端使用分頁顯示優惠券
2. 用 `v-if` 事先隱藏點擊後才會出現的選單，在組件內用一個 boolean `ref` 管理，點擊後設為 `true`

## 優惠券資料 >10MB 導致無法寫入 Local Storage

查了一下 Local Storage 支援的儲存大小是 5MB，而且看起來是網站中的所有儲存資料一起共用這 5MB，實測了一下如果我分別將兩個將近 3MB 的陣列同時往 Local Storage 放，兩者無法並存，但當我刪除了其中一筆之後，另外一筆原本可以放的就放得下了

目前網站是往縮減陣列中每個物件體積的方向設計，搭配 Mongo DB 的 `projection` 過濾不使用的值來達成這個目標，但未來若優惠券數量常態性地達到 1000+ 以上時，目前縮減物件體積的的寫法勢必要改變

## 強化搜尋時的效能

每張優惠券內都會有可選的餐點，如果搜尋時也要一併搜尋這些可選的餐點的話，則事先把這些餐點的名字取出來儲存成一個字串陣列並放進優惠券物件內以供搜尋使用，而非每次搜尋時才在 filter 內使用 `Array.map` 重新組一次這個字串陣列。

## 對資料庫資料的處理

寫這個專案的時候，對於 NoSQL 資料庫的操作也只有簡單的以 JSON 寫入跟讀取而已，由於針對[強化搜尋時的效能](#強化搜尋時的效能)這部分，需要在原先的物件內塞一個多餘的字串作為搜尋關鍵字時的依據，原本的做法是從資料庫拿到 JSON 後再在前端內使用 `Array.map` 處理，但接著發現 Mongo DB 有一種方式叫做聚合（aggregate），就這樣打開了我又需要專門寫一篇文章介紹它的契機。

### 使用 $Project 過濾不使用的值
由於肯德基本身的優惠券 JSON 有很多屬性，但實際上我們用不到那麼多，在從資料庫取出優惠券資料的時候，從 Network 看請求的資料時，728 張優惠券需要下載 12.8 MB 的 JSON，耗時 6.46 秒；透過操作 Mongo DB 時加入 $project 區塊，節省了將近 75% 的資料體積。

```typescript
collection.aggregate([
    {
        $project: {
        code: 1,
        // 'sets.Details': 1, // 如果要進一步投影這個 key 內部的值，這此行就不允許存在
        'sets.EndDate': 1,
        'sets.Name': 1,
        'sets.Original_Price': 1,
        'sets.Upa_Group': 1,
        'sets.UseReusableCup': 1,
        // 'sets.Details.MList': 1, // 如果要進一步投影這個 key 內部的值，這此行就不允許存在
        'sets.Details.MinCount': 1,
        'sets.Details.MaxCount': 1,
        'sets.Details.MList.AddPrice': 1,
        'sets.Details.MList.Name': 1,
        'sets.Details.MList.UseReusableCup': 1,
        },
    },
]).toArray();
```

經過處理後的資料，在陣列長度不變的情況下大小由 12.8 -> 3.2 MB，耗費的時間也由 6.46 -> 2.27 秒。

資料體積減少 75%，請求時間減少了 65%，我覺得很棒！！
若未來還需要進一步最佳化，可能就需要從屬性的長度下手了。

### 自動部署 & 在網站上顯示 Change Log（待補完）
由於專案是架設在 Vercel 上，目前只是簡單地在 Vercel 上選擇當分支有變更時就重新部署，但我希望以較正式的流程做這件事，

簡單的想法會是：
在 Github 上打 Tag 並寫好版本更新紀錄 -> 專案自動執行 `npm run build` 並推一份到 main 分支 -> Vercel 偵測 main 分支變更自動重新部署

並且還要在上面這個流程中根據對應的 Tag 顯示當前的網站版本號，並且有一個頁面可以顯示每個 Tag 與對應的更新紀錄

### 改善機器人的執行效率

由於肯德基網站優惠券涵蓋的編號大約是 10000 ~ 60000 之間，原先因為想要逐一在 Terminal 上顯示 Log 所以使用 for 迴圈搭配 await 按順序撈資料，接著再設定排程定時在 Github Actions 上每天自動執行一次腳本，但由於是按照順序打 API，因此每次的請求之間都會等待上一次的請求結束之後才執行，透過這種方式每次搜索完畢都需要花費 2+ 個小時的時間才能搜尋完畢。由於一開始只是小範圍測試（只撈幾十張優惠券）所以這個問題還不明顯，當範圍擴展到 50000 張優惠券之後，有一天突然就發現 Github Actions 一直告警執行失敗，才知道每天這樣更新的時間已經把免費帳號提供的 2000分鐘/月 的執行額度用完了。

由於這個問題攸關網站的主要業務，所以後續改以 `Promise.all` 同步撈資料，並且手動過濾有些優惠券根本不會出現的範圍後，現在大約只需要 20 秒就能執行完所有搜尋。

並且為了預防短時間內發出太多請求，在每個請求之間也加入一些延遲，假裝自己不是在 DDOS 肯德基網站XD。
