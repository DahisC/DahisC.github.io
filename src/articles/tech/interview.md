# 整理一些準備面試時發現的問題

## 常見問題

### 自我介紹
「哈囉，我是 Terry。講到我怎麼開始寫 Code 的，其實契機滿偶然的。

那時候我在電商平台當工讀生，每天要把產品上架到蝦皮、露天好幾個平台，重複動作超多。因為我真的很想早點下班（笑），就自己寫了一套自動化腳本，把原本要傳 5 次的東西改成一次搞定，直接省掉 80% 的時間。但也因為這樣，我發現自己對這種『用技術解決問題』的過程很有感，就開始利用省下來的時間鑽研前後端。

這幾年我的技術核心主要是 Vue 3、Nuxt 和 TypeScript。我自認技術好奇心滿重的，研究 JS 新語法或是摸索像 Nuxt 4 這種新東西對我來說不是壓力，反而像在玩遊戲。私底下我也會跑去學 NestJS 幫自己建立後端的架構邏輯，或是玩 ThreeJS 這種前端動畫。

在團隊裡，我應該算是一個滿好聊、滿 Chill 的人。但我對 Code 的整潔度稍微有一點強迫症（笑），我不喜歡那種隨便塞在一起的『麵條代碼』。開發的時候，我會習慣多花一點心思去做模組化，儘量不留技術債，因為我覺得把架構理清楚，後續維護才不會痛苦。

雖然寫程式很多時候是自己在動腦，但我其實很喜歡跟同事交流，特別是透過 Code Review 來互相學習。我覺得工程師之間如果能透過程式碼來認識彼此、看怎麼把邏輯寫得更漂亮，那種交流的感覺真的很棒。」

「……所以其實對我來說，這階段工程師不只是在寫功能，更是在提升整體的『開發體驗』。

最近我也在研究怎麼把 Claude 這些 AI 工具加進我平時的開發流程，看能不能讓重構或寫測試變得更直覺或是讓自己更有效率。

我覺得前端是一個跌代很快的領域，所以我也期許自己能夠保有一顆持續學習的心態面對接下來的職涯。

### 請分享一個你在開發中遇到「最大的技術挑戰」，你是如何解決的？

【情境與問題 (Situation/Task)】
「在我前公司（剛起步階段），開發節奏非常快，當時為了追求更新速度，專案累積了不少技術債。我入職時，業務邏輯已經變得相當複雜，雖然還沒正式上線，但我發現既有的程式碼存在很多重複邏輯，且沒有遵守單一事實來源 (SSOT) 的原則，導致後續功能極難擴充。」

【解決行動 (Action)】
「當時我被指派開發新功能，但我認為如果不解決根源問題，未來的開發成本會越來越高。

- 架構梳理： 我先花了一週時間完整梳理既有架構，並主動找技術主管溝通，分析現況並提議重構的可能性。

- 模組化實作： 得到認可後，我花了約三週時間進行重構。核心做法是將共用的 UI 抽成組件 (Components)，並利用 Vue 3 的特性將複雜的業務邏輯封裝成 Composables。

【最終結果 (Result)】
「重構後的程式碼順利通過測試上線。最直接的成效是，因為大幅提升了代碼的複用性，後續開發同類型新需求的時間節省了約 70%。這套邏輯架構一直到我離職時，公司團隊都還在持續沿用，這對我來說非常有成就感。」
